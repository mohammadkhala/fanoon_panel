# خطة تنفيذ ميزة تنبيه المخزون + API المنتجات العائدة + إعداد إظهار/إخفاء (نسخة 2)

**الهدف:**  
1. إظهار تنبيه المخزون الأدنى في الهيدر مثل تنبيه الطلبات (أيقونة + رقم) مع **قائمة منسدلة** تحتوي أسماء المنتجات التي قاربت على الانتهاء.  
2. توفير **API** للموقع/التطبيق تعرض فيها **المنتجات التي رجعت للمخزون**.  
3. في **إعدادات العمل (ecom-setup)** إضافة خيار: **إظهار أو إخفاء** (قسم/API المنتجات العائدة) عن العملاء.

**ملاحظة:** لا يُنفذ أي كود حتى تقول "ابدأ".

---

## تحليل جدول المنتجات في قاعدة البيانات

من الـ migrations والـ Model المعنية بالمنتج والمخزون:

| العمود | النوع | الوصف |
|--------|--------|--------|
| `id` | bigint (PK) | معرف المنتج |
| `name` | string | اسم المنتج (قد يكون عبر جدول الترجمات) |
| `image` | text (nullable) | صور المنتج (JSON أو مصفوفة) |
| `price` | float | السعر |
| `tax` | float | الضريبة |
| `discount` | float | الخصم |
| `status` | integer | الحالة (مفعّل/معطّل) |
| `total_stock` | bigInteger, default 0 | **المخزون الحالي** |
| `minimum_stock_alert` | unsignedInteger, nullable | **حد تنبيه المخزون الأدنى** (لكل منتج؛ إن كان null يُستخدم الإعداد العام) |
| `unit` | string, nullable | وحدة القياس (pc وغيرها) |
| `variations` | JSON (في أغلب الأنظمة) | تشكيلات المنتج (قد تحتوي stock لكل تشكيلة) |
| أعمدة أخرى | — | created_at, updated_at, branch_id, category_id, إلخ |

**منطق "مخزون منخفض" (موجود في النظام):**  
- `Product::scopeLowStock($defaultAlert)`: منتجات حيث `total_stock <= minimum_stock_alert` أو (إن كان minimum_stock_alert null) `total_stock <= default_minimum_stock_alert` (من business_settings، افتراضي 5).  
- لا يُحتسب فيها المخزون السالب كـ "قارب على الانتهاء" فقط؛ المنطق الحالي يضم أيضاً من انتهى مخزونهم إذا كانوا ≤ الحد.

هذا الجدول كافٍ لتنفيذ: (1) قائمة المنتجات ذات المخزون المنخفض في الهيدر، (2) تحديد "منتجات عائدة للمخزون" إذا أضفنا تتبعاً بسيطاً (انظر المرحلة 2).

## المرحلة 1: تنبيه المخزون في الهيدر (أيقونة + قائمة منسدلة)

### 1.1 الشكل المطلوب (مثل تنبيه الطلبات لكن مع قائمة منسدلة)

- **الموقع:** الهيدر في لوحة التحكم، بجانب أيقونة الطلبات (أو بعدها).
- **العنصر:**
  - **أيقونة** مناسبة (مثلاً `tio-package-outlined` أو أيقونة مخزون).
  - **شارة (badge)** تعرض **عدد** المنتجات ذات المخزون المنخفض (نفس منطق `Product::lowStock()`).
  - عند النقر: **قائمة منسدلة (dropdown)** وليس انتقالاً لصفحة أخرى، تحتوي:
    - عناصر القائمة: **أسماء المنتجات** التي مخزونها منخفض (مع عرض المخزون الحالي إن أمكن، اختياري).
    - رابط في أسفل القائمة: **"عرض الكل"** يوجّه إلى `admin/product/list?stock_filter=low_stock`.

### 1.2 التحقق من الهيدر الحالي (الطلبات)

- الملف: `resources/views/layouts/admin/partials/_header.blade.php`.
- عنصر الطلبات الحالي:
  - `<div class="hs-unfold">` ثم `<a class="js-hs-unfold-invoker btn btn-icon btn-ghost-secondary rounded-circle" href="{{ route('admin.orders.list',['status'=>'all']) }}">` مع أيقونة و `<span class="btn-status btn-status-danger">العدد</span>`.
  - **ملاحظة:** الرابط الحالي للطلبات يفتح الصفحة مباشرة وليس dropdown. المطلوب لعنصر المخزون: **dropdown** يحتوي قائمة أسماء المنتجات + "عرض الكل".

### 1.3 التنفيذ المقترح

- **نفس الملف:** `_header.blade.php`.
- إضافة عنصر جديد (مثلاً بعد عنصر الطلبات):
  - غلاف `hs-unfold` مع زر يفتح قائمة منسدلة (`data-hs-unfold-options` يشير إلى `#lowStockNavbarDropdown` أو ما شابه).
  - على الزر: أيقونة + `<span class="btn-status btn-status-warning">عدد المنتجات منخفضة المخزون</span>`.
  - **محتوى القائمة المنسدلة:**  
    `#lowStockNavbarDropdown` يحتوي:
    - عنوان قصير (مثلاً: "منتجات مخزون منخفض").
    - حلقة على **قائمة منتجات منخفضة المخزون** (يُمرَّر من Controller أو عبر Livewire/JS؛ الأبسط: تمريرها من Layout أو View Composer).
    - كل عنصر: اسم المنتج + (اختياري) `total_stock`، وربط بـ `route('admin.product.view', $product->id)`.
    - في الأسفل: `<a href="{{ route('admin.product.list', ['stock_filter' => 'low_stock']) }}">عرض الكل</a>`.

### 1.4 مصدر البيانات للقائمة المنسدلة

- الخيار **الأبسط:** استخدام **View Composer** يربط بـ `View::composer('layouts.admin.partials._header', ...)` ويمرر متغيراً مثل `$lowStockProducts = Product::lowStock()->take(10)->get()` (أو 15) حتى لا تطول القائمة.
- بديل: تمرير العدد فقط في الـ Composer وعرض القائمة عبر طلب AJAX عند فتح الـ dropdown (أكثر تعقيداً).
- **العدد على الشارة:** من نفس الـ Composer: `$lowStockCount = Product::lowStock()->count()`.

### 1.5 الملفات المتأثرة (المرحلة 1)

| الإجراء | الملف/المسار |
|---------|----------------|
| إنشاء أو تعديل | `app/Providers/ViewServiceProvider.php` أو `AppServiceProvider` (View Composer للهيدر) |
| تعديل | `resources/views/layouts/admin/partials/_header.blade.php` (أيقونة + badge + dropdown بقائمة الأسماء + عرض الكل) |
| إضافة ترجمات | `resources/lang/ar/messages.php` و `en/messages.php` (عنوان القائمة، "عرض الكل"، إن لزم) |

## المرحلة 2: API المنتجات العائدة للمخزون

### 2.1 المقصود بـ "المنتجات التي رجعت على المخزون"

- منتجات كانت **خارج المخزون** (`total_stock <= 0`) ثم أصبح **لديها مخزون** (`total_stock > 0`).
- لتمييز "عادت الآن" من "كانت دائماً متوفرة" نحتاج **تتبعاً بسيطاً**.

### 2.2 تتبع "عاد للمخزون" (اقتراح)

- **خيار (أ):** عمود في جدول `products`:  
  `back_in_stock_at` (timestamp, nullable).  
  - عند أي تحديث يزيد `total_stock` من 0 إلى قيمة > 0: نحدّث `back_in_stock_at = now()`.  
  - الـ API ترجع منتجات حيث `back_in_stock_at` غير null (ويفضل ترتيبها حسب `back_in_stock_at` تنازلياً).  
  - بعد فترة أو عند "مسح" من الإدارة يمكن إعادة `back_in_stock_at = null` حتى لا تبقى في القائمة للأبد.

- **خيار (ب):** جدول منفصل `product_stock_events`: (id, product_id, event_type ['back_in_stock'], created_at).  
  - عند عودة المخزون نُدخل سجلاً. الـ API تقرأ آخر حدث "back_in_stock" لكل product.  
  - أنسب إذا أردت تاريخاً كاملاً للأحداث لاحقاً.

**التوصية للخطة الحالية:** خيار (أ) لتبسيط التنفيذ.

### 2.3 تحديث `back_in_stock_at`

- في كل مكان يُحدَّث فيه `products.total_stock` (بعد خصم أو إضافة):  
  إذا كانت القيمة **الجديدة** > 0 وكانت **القديمة** <= 0، نضع `back_in_stock_at = now()`.  
- عند خصم المخزون (طلب/إلغاء تعديل): إذا أصبح `total_stock <= 0` يمكن ترك `back_in_stock_at` كما هو أو عدم المساس به (المنتج لن يظهر في API "العائدة" لأنه غير متوفر).

### 2.4 تصميم الـ API

- **المسار المقترح:** `GET /api/v1/products/back-in-stock` (أو تحت prefix مثل `api/v1` حسب هيكل المشروع).
- **الاستجابة:** قائمة منتجات (مع الحقول المناسبة للتطبيق: id, name, image, price, total_stock, إلخ) حيث:
  - `total_stock > 0`
  - `back_in_stock_at` is not null
  - (اختياري) ترتيب حسب `back_in_stock_at` تنازلياً.
- **التحكم بالظهور للعملاء:** إذا كان إعداد "إظهار المنتجات العائدة للعملاء" **معطلاً** (انظر المرحلة 3)، يُرجع الـ API قائمة فارغة أو رسالة مناسبة (بدون بيانات المنتجات).

### 2.5 الملفات المتأثرة (المرحلة 2)

| الإجراء | الملف/المسار |
|---------|----------------|
| إنشاء migration | إضافة عمود `back_in_stock_at` لجدول `products` (أو جدول أحداث إن اخترت خيار ب) |
| تعديل | كل مكان يحدّث `total_stock` (OrderController API/Admin/Branch، POS، ProductController عند التعديل اليدوي للمخزون): إضافة منطق تحديث `back_in_stock_at` |
| إنشاء/تعديل | Controller للـ API (مثلاً في Api\V1) + Route لـ GET products/back-in-stock |
| (اختياري) | توثيق الـ API للمبرمج |

## المرحلة 3: إعداد إظهار/إخفاء "المنتجات العائدة" للعملاء (إعدادات العمل)

### 3.1 المطلوب

- في صفحة **إعدادات العمل (ecom-setup)** إضافة خيار:
  - **"إظهار المنتجات العائدة للمخزون للعملاء"** (أو "إتاحة API/قسم المنتجات العائدة للعملاء").
  - قيمتان: يظهر / لا يظهر (مثل مفتاح أو زر تبديل).

### 3.2 التخزين

- في `business_settings`: مفتاح جديد، مثلاً `show_back_in_stock_to_customers` (قيمة 1 = يظهر، 0 أو غير موجود = لا يظهر).

### 3.3 الاستخدام

- في **API** المنتجات العائدة: قبل إرجاع النتائج، التحقق من `Helpers::get_business_settings('show_back_in_stock_to_customers')`.  
  - إذا معطّل: إرجاع مصفوفة فارغة أو رسالة توضح أن الميزة غير مفعّلة، دون كشف بيانات المنتجات.

### 3.4 الملفات المتأثرة (المرحلة 3)

| الإجراء | الملف/المسار |
|---------|----------------|
| تعديل | `resources/views/admin-views/business-settings/ecom-setup.blade.php` (إضافة قسم/زر التبديل) |
| تعديل | Controller الذي يحفظ إعدادات ecom-setup (حفظ المفتاح الجديد في `business_settings`) |
| تعديل | Controller الـ API للمنتجات العائدة (قراءة الإعداد واتخاذ قرار الإرجاع) |
| إضافة ترجمات | مفاتيح للعرض في إعدادات العمل (عربي/إنجليزي) |

## ترتيب التنفيذ المقترح

1. **المرحلة 1:** View Composer للهيدر → تعديل الهيدر (أيقونة + badge + dropdown بأسماء المنتجات + رابط "عرض الكل").  
2. **المرحلة 2:** Migration لـ `back_in_stock_at` (أو جدول الأحداث) → تحديث كل أماكن تغيير `total_stock` → إنشاء API للمنتجات العائدة.  
3. **المرحلة 3:** إضافة الخيار في ecom-setup وحفظه في `business_settings` → ربط الـ API بهذا الإعداد.

## ملخص الجداول والإعدادات

- **جدول المنتجات (products):** الأعمدة ذات الصلة بالميزة: `total_stock`, `minimum_stock_alert`. يُضاف لها (في الخطة الحالية) `back_in_stock_at` إن تم اعتماد خيار (أ).  
- **إعدادات الأعمال (business_settings):** مفتاح `default_minimum_stock_alert` (موجود)، مفتاح جديد `show_back_in_stock_to_customers` (للمرحلة 3).
