## 3. ترتيب التنفيذ المقترح

### الأسبوع 1 — أمان واستقرار (قبل التسليم)

```
اليوم 1–2: مراجعة أمان الاستعلامات
├── البحث عن DB::raw و whereRaw
├── التأكد من استخدام bindings أو escape
└── إصلاح أي ثغرات محتملة

اليوم 3: حد API
├── إضافة max limit (مثلاً 50) في API v1 و v2
└── إرجاع خطأ واضح عند تجاوز الحد

اليوم 4: CORS
├── مراجعة config/cors.php
└── التأكد من عدم استخدام * في الإنتاج

اليوم 5: اختبار شامل
├── تسجيل الدخول (Admin)
├── إضافة/تعديل منتج (وسوم، منتجات ذات صلة)
├── عرض طلب وتغيير الحالة (سجل التغييرات)
├── إنشاء طلب من API
└── تصدير الطلبات
```

### الأسبوع 2 — تحسينات سريعة (اختياري)

```
اليوم 1–2: كاش الطلبات
├── تخزين مؤقت لعدد الطلبات (مثلاً 1 دقيقة)
└── تحديث عند تغيير حالة طلب

اليوم 3: توحيد المصطلحات (إن وُجد وقت)
└── راجع PLAN-TERMINOLOGY-UNIFICATION.md
```

## 4. قائمة تحقق قبل التسليم

- [ ] مراجعة أمان الاستعلامات (DB::raw / whereRaw)
- [ ] حد أقصى لـ limit في API
- [ ] CORS آمن في الإنتاج
- [ ] اختبار تسجيل الدخول
- [ ] اختبار إضافة منتج (وسوم + منتجات ذات صلة)
- [ ] اختبار عرض طلب وسجل التغييرات
- [ ] اختبار API أساسي (products, orders)
- [ ] التأكد من عدم وجود أخطاء في الـ logs
- [ ] بيانات تجريبية مناسبة (BaitPaitSeeder)

## 5. ملخص تنفيذي

| المرحلة | المهام | الوقت المقدر |
|---------|--------|---------------|
| **ضروري** | أمان + حد API + اختبار | 1 أسبوع |
| **مُفضّل** | CORS + كاش الطلبات | 2–3 أيام |
| **لاحقاً** | توحيد مصطلحات، Redis، ميزات إضافية | بعد النسخة الأولى |

**التوصية:** التركيز على المهام الثلاث الأولى (أمان، حد API، اختبار) ثم التسليم. باقي التحسينات يمكن تنفيذها في تحديثات لاحقة.
