# خطة تحسين النظام — منافسة ووردبريس وووكومرس

> **الهدف:** تنظيف النظام وإضافة تحسينات تجعله أفضل من ووردبريس وووكومرس في نقاط محددة.

---

## 1. التنظيف (Cleanup)

### 1.1 إزالة الكود غير المستخدم

| العنصر | الملف | الإجراء |
|--------|-------|---------|
| LsLibController | `app/Http/Controllers/Api/V2/LsLibController.php` | **حذف** — المسار معطّل لكن الـ Controller ما زال موجوداً (ثغرة سابقة) |
| API V2 | `routes/api/v2/api.php` | مراجعة — إن كان فارغاً، إما حذفه أو توثيقه |
| Modules فارغة | `Modules/` | إما استخدامها للإضافات أو إزالة `nwidart/laravel-modules` إن لم تُستعمل |
| مسارات Install/Update | `RouteServiceProvider` | مراجعة — إن كانت معطّلة، توثيق أو إزالة |

### 1.2 توحيد المصطلحات

| الحالي | المقترح | السبب |
|--------|----------|-------|
| `restaurant` / `get-restaurant-data` | `store` / `get-store-data` | النظام تجارة إلكترونية وليس مطاعم |
| `restaurant_name` | `shop_name` أو `store_name` | توافق مع السياق |

> **خطة تفصيلية:** راجع `docs/PLAN-TERMINOLOGY-UNIFICATION.md` قبل التنفيذ.

### 1.3 تحسين جودة الكود

- **SQL Injection:** مراجعة كل استخدامات `DB::raw` و `whereRaw` — التأكد من أن المتغيرات مُهربّة
- **N+1 في الشريط الجانبي:** استبدال عدة `Order::count()` باستعلام واحد أو كاش

## 2. تحسينات الأداء

| التحسين | الوصف | الأولوية |
|---------|-------|----------|
| Redis للكاش | استخدام Redis بدلاً من File في الإنتاج | عالية |
| كاش عدد الطلبات | تخزين مؤقت لعدد الطلبات في الشريط الجانبي (مثلاً 1 دقيقة) | عالية |
| حد API | تحديد أقصى `limit` و `offset` (مثلاً 50) لتجنب استعلامات ثقيلة | متوسطة |
| تحميل كسول للصور | `loading="lazy"` للصور في القوائم | متوسطة |
| ضغط الاستجابة | تفعيل Gzip في الخادم | متوسطة |

## 3. تحسينات الأمان

| التحسين | الوصف |
|---------|-------|
| إزالة LsLibController | حذف الملف بالكامل |

## 4. تحسينات تجربة المستخدم (UX)

### 4.1 لوحة التحكم

| التحسين | الوصف |
|---------|-------|
| إحصائيات فورية | تحديث الأرقام دون إعادة تحميل الصفحة |
| تنبيهات ذكية | تنبيه عند انخفاض المخزون، طلبات معلقة طويلة |
| بحث موحّد | بحث عام في الطلبات والمنتجات والعملاء من مكان واحد |

### 4.2 المنتجات

| الميزة (مثل ووكومرس) | الحالة الحالية | التحسين |
|----------------------|----------------|---------|
| وسوم المنتجات | ✅ مكتمل | — |
| فلاتر متقدمة | أساسية + سعر + مخزون + وسم | فلتر التقييم فقط |
| بحث متقدم | أساسي + نطاق سعري | بحث بالخصائص (اختياري) |
| منتجات ذات صلة | ✅ مكتمل | — |

### 4.3 الطلبات

| التحسين | الوصف | الحالة |
|---------|-------|--------|
| سجل التغييرات | تسجيل من غيّر حالة الطلب ومتى | ✅ مكتمل |
| إشعارات فورية | إشعار عند طلب جديد (صوت + تنبيه) | جزئي |
| طباعة جماعية | طباعة فواتير عدة طلبات دفعة واحدة | ❌ لم يُنفّذ |

### 4.4 العملاء

| التحسين | الوصف | الحالة |
|---------|-------|--------|
| شارات مراجعة | "عميل موثوق" أو "مشتري مؤكد" للتقييمات | ❌ لم يُنفّذ |
| تقسيم العملاء | تقسيم حسب القيمة، التكرار، آخر شراء | ❌ لم يُنفّذ |
| تصدير القوائم | تصدير قائمة العملاء إلى Excel | ❌ لم يُنفّذ |

## 6. تحسينات مقترحة حسب الأولوية

### المرحلة 1 — تنظيف وأمان (أسبوع)

1. حذف `LsLibController.php`
2. مراجعة `DB::raw` و `whereRaw`
3. إضافة حد أقصى لـ `limit` في API

### المرحلة 2 — أداء (أسبوعان)

1. كاش لعدد الطلبات في الشريط الجانبي
2. تفعيل Redis للكاش (إن أمكن)
3. تحميل كسول للصور

### المرحلة 3 — UX (شهر)

1. وسوم المنتجات ✅
2. فلاتر متقدمة للمنتجات (جزئي — بقي فلتر التقييم)
3. سجل تغييرات الطلبات ✅
4. بحث موحّد في لوحة التحكم ❌

### المرحلة 4 — ميزات جديدة (شهرين)

1. توثيق API (OpenAPI/Swagger)
2. Webhooks للأحداث (طلب جديد، تغيير حالة)
3. تقارير وتحليلات متقدمة
4. اشتراكات/طلبات متكررة (إن لزم)

## 9. ملخص تنفيذي

| المحور | الإجراء |
|--------|---------|
| **تنظيف** | حذف LsLibController، مراجعة كود غير مستخدم |
| **أمان** | مراجعة raw queries، CORS، throttling |
| **أداء** | كاش، Redis، حد API |

| **تنافسية** | التركيز على: خفة، عربية، تطبيق جوال، سهولة |

النظام حالياً قوي في الأساس. التركيز على التنظيف والأداء والـ UX سيجعله منافساً جداً لووكومرس في السوق العربي.
