# تقرير تحليل التصنيفات والتصنيفات الفرعية

> **الهدف:** هل يمكن إضافة أكثر من مستوى فرعي (أب → ابن → ابن الابن)؟  
> **العميل يحتاج:** إما (أب + ابن) أو (أب + ابن + ابن ابن) حسب الحاجة.

---

## 1. هيكل جدول التصنيفات

### 1.1 جدول `categories`

| العمود | النوع | الوصف |
|--------|-------|-------|
| `id` | bigint | المفتاح الأساسي |
| `name` | varchar(255) | اسم التصنيف |
| `parent_id` | bigint (default 0) | **الأب** — 0 للجذر، أو id التصنيف الأب |
| `position` | int (default 0) | 0=جذر، 1=فرعي (يُستخدم في الواجهة للتمييز) |
| `status` | boolean | نشط/معطّل |
| `image`, `banner_image` | varchar | الصور |
| `is_featured` | tinyint | مميز أم لا |

### 1.2 العلاقة الهرمية

```
parent_id = 0  →  تصنيف جذر (أب)
parent_id = X  →  ابن للتصنيف X
```

**الخلاصة:** العمود `parent_id` يدعم **هرمية غير محدودة** — أي تصنيف يمكن أن يكون أباً لأي تصنيف آخر.

---

## 2. ما يدعمه النظام حالياً

### 2.1 قاعدة البيانات

| العنصر | الحالة |
|--------|--------|
| هيكل `parent_id` | ✅ يدعم أب → ابن → ابن ابن → ... |
| لا قيود على العمق | ✅ يمكن إضافة أي عدد من المستويات |

### 2.2 منطق المنتجات (`CategoryLogic::all_products`)

> ملاحظة: كان المنطق سابقاً يجمع المنتجات من **3 مستويات** فقط، وتم تحديثه لاحقاً ليشمل **كل الأحفاد بأي عمق**.

```php
// CategoryLogic.php — all_products($id)
$cate_ids[] = (int)$id;                    // المستوى 1: التصنيف نفسه
foreach (child($id) as $ch1) {
    $cate_ids[] = $ch1['id'];              // المستوى 2: الأبناء
    foreach (child($ch1['id']) as $ch2) {
        $cate_ids[] = $ch2['id'];          // المستوى 3: أبناء الأبناء
    }
}
```

**الخلاصة:** منطق المنتجات أصبح يدعم أب + ابن + ابن ابن + … (أي عمق).

### 2.3 API

| Endpoint | الوظيفة |
|----------|---------|
| `GET /api/v1/categories` | التصنيفات الجذرية (position=0) |
| `GET /api/v1/categories/childes/{id}` | الأبناء المباشرون لأي تصنيف |

**الخلاصة:** الـ API يدعم التكرار — يمكن طلب `childes` لأي تصنيف (جذر أو فرعي) للحصول على المستوى التالي.

### 2.4 المنتجات (`category_ids`)

المنتج يخزن `category_ids` كـ JSON، مثل:

```json
[{"id": "5", "position": 1}, {"id": "12", "position": 2}]
```

يمكن ربط المنتج بأي تصنيف (جذر أو فرعي أو فرعي فرعي) — لا قيد على المستوى.

## 3. القيود الحالية (واجهة الإدارة)

### 3.1 إضافة تصنيف فرعي

كان في `sub-index.blade.php`:

```php
<select name="parent_id">
    @foreach(Category::where(['position'=>0])->get() as $category)
        <option value="{{ $category['id'] }}">{{ $category['name'] }}</option>
    @endforeach
</select>
```

**المشكلة (سابقاً):** قائمة الأب تعرض **فقط التصنيفات الجذرية** (position=0). لا يمكن اختيار تصنيف فرعي كأب لإنشاء ابن ابن.

**الحل (تم):** أصبحت قائمة الأب هرمية وتعرض كل التصنيفات (جذر + فرعي + …) مما يسمح بإنشاء ابن ابن.

### 3.2 عرض التصنيفات الفرعية

- `index`: يعرض التصنيفات الجذرية (position=0)
- `subIndex`: يعرض كل التصنيفات الفرعية (position=1) في قائمة واحدة مسطحة

لا يوجد تمييز بين ابن وابن ابن في الواجهة — كلاهما position=1.

---

## 4. التوصيات لتفعيل (أب + ابن) أو (أب + ابن + ابن ابن)

### 4.1 الخيار أ: توسيع قائمة الأب (الأبسط)

**التعديل:** في نموذج إضافة التصنيف الفرعي، عرض **كل التصنيفات التي يمكن أن تكون أماً** (جذر + فرعي):

```php
// بدلاً من: where(['position'=>0])
Category::where('status', 1)->orderBy('parent_id')->orderBy('name')->get()
```

أو بشكل هرمي:

```php
// جذور + فرعيات (لاختيار أب فرعي فرعي)
Category::whereIn('position', [0, 1])->get()
```

**النتيجة:** يمكن اختيار تصنيف فرعي كأب، وبالتالي إنشاء ابن ابن.

### 4.2 الخيار ب: عمق ديناميكي (position)

- جذر: `position = 0`
- ابن: `position = 1`
- ابن ابن: `position = 2`

**التعديل:** عند الإضافة، حساب `position` تلقائياً من عمق الأب:

```php
$parent = Category::find($request->parent_id);
$position = $parent ? $parent->position + 1 : 0;
```

### 4.3 الخيار ج: صفحة منفصلة لـ "ابن ابن"

إضافة route و view جديد لـ "تصنيف فرعي فرعي" مع قائمة أب تعرض الجذور والفرعيات.

## 5. ملخص تنفيذي

| السؤال | الإجابة |
|--------|---------|
| **هل قاعدة البيانات تدعم أب → ابن → ابن ابن؟** | ✅ نعم — `parent_id` يدعم ذلك بالكامل |
| **هل منطق المنتجات يدعم 3 مستويات؟** | ✅ نعم — `CategoryLogic::all_products` يجمعها |
| **هل يمكن للعميل استخدام أب+ابن فقط؟** | ✅ نعم — يعمل كما هو |
| **هل يمكن للعميل استخدام أب+ابن+ابن ابن؟** | ✅ نعم — بعد تعديل الواجهة والمنطق |

### ما تم تنفيذه

1. ✅ **توسيع قائمة الأب** في إضافة/تعديل التصنيف الفرعي لتشمل كل التصنيفات بشكل هرمي.
2. ✅ **منع الحلقات** عند تعديل الأب (التصنيف لا يمكن أن يصبح ابنًا لنفسه أو لأحد أبنائه).
3. ✅ **تحديث `CategoryLogic::all_products`** ليشمل كل الأحفاد بأي عمق.

## 7. مثال بيانات (أب → ابن → ابن ابن)

لتجربة عرض البيانات في الصفحات، شغّل:

```bash
php artisan db:seed --class=CategoryHierarchyExampleSeeder
```

**البيانات المضافة:**
- **أب:** سجائر إلكترونية (id=1)
- **ابن:** سوائل نيكوتين (id=2، parent_id=1)
- **ابن ابن:** سوائل نيكوتين ملح (parent_id=2)
- **منتج:** سائل ملح نيكوتين 20mg — مرتبط بالمسار الكامل

**أين تظهر:**
- لوحة الإدارة → التصنيفات → التصنيفات الفرعية: تظهر الثلاثة (ابن وابن ابن في القائمة)
- لوحة الإدارة → إضافة تصنيف فرعي: قائمة الأب تعرض هرمياً (سجائر إلكترونية — سوائل نيكوتين)
- API `GET /categories/childes/1` → سوائل نيكوتين
- API `GET /categories/childes/2` → سوائل نيكوتين ملح
- API `GET /categories/products/1/all` → يشمل منتج سائل ملح نيكوتين 20mg
